### 2. Unescapte Instanz-Namen in status.js (XSS-Risiko)

**Datei:** `admin-interface/static/js/status.js`, Zeilen 116–122

```javascript
<button onclick="instanceControl('${a.inst}','restart')">
<button onclick="instanceControl('${a.inst}','stop')">
<button onclick="instanceForceStop('${a.inst}')">
<span style="font-weight:500;">${dot} ${a.inst}</span>
```

**Problem:** `a.inst` (Instanz-Name aus `/api/agent-health/{inst}`) wird direkt in `innerHTML` eingebaut ohne `escAttr()` / `escHtml()`. Instanz-Namen kommen aus `INSTANCES` (Jinja2 in index.html), der Wert stammt aus `config.json`. Ein Instanz-Name mit `'` oder `">` könnte onclick-Attribute brechen oder Code injizieren.

**Risiko:** Moderat — der Angriffspfad erfordert Admin-Zugang zur config.json. Aber: Regel aus CLAUDE.md „immer escAttr() in onclick-Attributen" ist hier verletzt.

**Empfehlung:** `${a.inst}` → `${escAttr(a.inst)}` in onclick-Attributen, `${escHtml(a.inst)}` im Text.

### 3. Unescapte Error-Messages in innerHTML (XSS-Risiko)

**Datei:** `admin-interface/static/js/status.js`, Zeile 182

```javascript
grid.innerHTML = `<div class="status-card">...<div>${e.message}</div></div>`;
```

**Problem:** `e.message` (JavaScript Error-Objekt) wird direkt in innerHTML eingebaut. Bei einem manipulierten API-Response könnte der Fehlermeldungstext HTML enthalten. Gleicher Fehler in `config.js` Zeile 693, 909 (e.message ohne escHtml in textContent — dort unkritisch da textContent, aber inkonsistent).

**Empfehlung:** `${e.message}` → `${escHtml(e.message)}` überall wo innerHTML verwendet wird.

## Warnung (diese Woche fixen)

### 4. /api/wa-proxy/ weiterhin ohne Auth

**Datei:** `admin-interface/main.py`, Zeile 215
**Status:** Bekannt aus gestrigem Audit (Finding #4), nicht behoben.

```python
_AUTH_EXEMPT_PREFIXES = ("/static/", "/ws/", "/api/wa-proxy/", "/api/companion/")
```

**Problem:** Der wa-proxy-Endpunkt ist vollständig ohne Auth erreichbar. Jeder mit Netzwerkzugriff auf Port 8080 kann Nachrichten an beliebige Agenten senden und so Antworten provozieren oder den Agenten-Kontext manipulieren.

**Empfehlung:** `BRIDGE_SECRET` (bereits in `routers/deps.py` definiert) auch für `/api/wa-proxy/` prüfen — die WA-Bridge sendet `X-Bridge-Token`, der bereits für `/api/whatsapp-config` und `/api/users/whatsapp-lid` geprüft wird.

---

### 5. /api/companion/ exempt ohne Token-Prüfung (konzeptionelle Inkonsistenz)

**Datei:** `admin-interface/main.py`, Zeile 215
**Status:** Bekannt aus gestrigem Audit (Finding #7), nicht behoben.

**Problem:** `/api/companion/` ist im Auth-Exempt-Prefix — der Companion-Token wird zwar korrekt in den Endpunkt-Funktionen geprüft (`verify_companion_token`), aber die Middleware bypasst die Prüfung ganz. Bei einem neuen `/api/companion/`-Endpunkt, der die Token-Prüfung vergisst, würde er ungeschützt sein.

**Empfehlung:** `/api/companion/` aus `_AUTH_EXEMPT_PREFIXES` entfernen und stattdessen in der Middleware den Companion-Token als gültige Auth akzeptieren (wie es für Bearer-Token bei `/api/wa-proxy/` nicht gemacht wird). Alternativ: `/api/companion/ping` und `/api/companion/sso` explizit in `_AUTH_EXEMPT_EXACT` listen.
