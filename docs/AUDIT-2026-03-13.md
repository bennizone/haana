## Kritisch (sofort fixen)

### 1. Admin-Token bei jedem Neustart im Log (`auth.py:136`)
**Datei:** `admin-interface/auth.py`, Zeile 136
```python
print(f"HAANA Admin Token: {token}", flush=True)
```
Bei **jedem** Neustart im Standalone-Modus wird der Token im Klartext in stdout geloggt.
Auf `.12` ist er in `docker logs haana-admin-interface-1` sichtbar.
Jeder mit Docker-Log-Zugriff kann sich als Admin authentifizieren.
**Zeile 44** (`logger.info(...)`) feuert dagegen nur bei erstmaliger Generierung — akzeptabler.
**Fix:** Startup-Log entweder entfernen oder auf WARN-Level mit ersten/letzten 4 Zeichen maskiert ausgeben.

### 3. mem0 `result={'results': []}` bei jedem Memory-Write (Prod-Beobachtung)
**Log:** `haana-instanz-benni-1`, alle Memory-Save-Events der letzten Stunde:
```
mem0.memory.main: Total existing memories: 0
core.memory: Memory gespeichert | result={'results': []}
```
mem0's `add()` gibt konsistent leere Ergebnisse zurück. Das kann bedeuten:
- (a) mem0 klassifiziert alle Nachrichten als nicht-merkwürdig (kein Fact-Content) — normales Verhalten
- (b) Die Suche innerhalb von mem0 findet nie vorhandene Memories zur Deduplication → Neue Entries werden nie wirklich gespeichert

Aktion: Im Admin-Interface die Memory-Statistiken prüfen (Memory-Tab → Stats). Falls `benni_memory` wächst, ist (a) korrekt. Falls die Anzahl stagniert, ist (b) ein Bug.

## Warnung (bald fixen)

### 4. `/api/wa-proxy/` vollständig ohne Auth
**Datei:** `admin-interface/main.py`, Zeile 215
```python
_AUTH_EXEMPT_PREFIXES = ("/static/", "/ws/", "/api/wa-proxy/", "/api/companion/")
```
Der WA-Proxy-Endpunkt (wird vom WhatsApp-Bridge-Container genutzt) ist ohne jede Auth erreichbar. Jeder mit Netzwerkzugriff auf Port 8080 kann Nachrichten an beliebige Agenten senden.
Da das Admin-Interface im LAN läuft und Port 8080 nicht nach außen exposed ist, ist das Risiko gering — aber nicht null.
**Fix:** Companion-Token auch für `/api/wa-proxy/` als Bearer prüfen (WA-Bridge hat den Token bereits konfiguriert oder kann ihn aus Config lesen).

### 5. WhatsApp Bridge: 503-Fehler auf Prod in den letzten Stunden
**Log:** `haana-whatsapp-bridge-1`, mehrfach:
```
Fehler bei Agent-Anfrage | agent: ".../api/wa-proxy/benni" | err: "Agent HTTP 503: Agent 'benni' nicht erreichbar"
```

## Technische Schuld (irgendwann)

### 7. `/api/companion/` exempt ohne interne Token-Prüfung
`/api/companion/` ist im Auth-Exempt-Prefix-Match. Die Middleware prüft den Companion-Bearer-Token erst **nach** dem Prefix-Check — also nie für `/api/companion/`.
Der Companion sendet seinen Token korrekt, aber er wird nicht geprüft.
Aktuell kein direktes Sicherheitsrisiko (Endpunkte tun nur Read/Register-Operationen), aber konzeptionell inkonsistent.

### 8. `localhost:11434` als Ollama-Fallback hardcodiert (mehrfach)
In `core/ollama_compat.py` (2×), `core/process_manager.py` (2×), `admin-interface/routers/config.py` (3×).
Das ist ein akzeptabler Default, aber der selbe Default-Wert ist 7× im Code — eine Konstante `DEFAULT_OLLAMA_URL` in einem `core/constants.py` wäre wartbarer.

### 9. `haana-plan-v7-final.md` fehlt
MEMORY.md referenziert diese Datei als Planungsdokument — sie existiert nicht im Repo. Entweder wurde sie nie angelegt oder gelöscht. Falls noch relevant: anlegen oder Referenz aus Memory entfernen.

### 10. LOGBOOK.md (EN) parallel zu LOGBUCH.md (DE)
`docs/LOGBOOK.md` und `docs/LOGBUCH.md` existieren nebeneinander. Unklar ob beide aktiv gepflegt werden. Duplizierung erzeugt Wartungsaufwand.

### 11. Bekannter Bug: `setup-token` schreibt Token manchmal nur nach stdout
Steht seit Längerem in MEMORY.md als bekanntes Problem. `claude_auth.py` hat bereits einen Workaround (stdout-Fallback-Parsing). Der Root-Cause in `setup-token` sollte langfristig untersucht werden.
